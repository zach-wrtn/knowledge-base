## How to Use

Evaluator는 평가 시작 시:
1. 현재 스프린트의 `evaluation/criteria.md` 로드
2. `knowledge-base/rubrics/` 디렉토리에서 **superseded_by가 null인 최신 vN** 로드
3. 두 문서의 clause를 합쳐 평가에 적용. 충돌 시 sprint 로컬 criteria가 우선.

## Clauses

### C1. Verdict Bands (베이스라인)

| 판정 | 조건 |
|------|------|
| PASS   | Critical 0, Major 0 |
| ISSUES | Critical 0, Major 1+ |
| FAIL   | Critical 1+, 또는 Major 3+ |

### C2. Severity 정의 (베이스라인)

- **critical**: 기능 불가, 데이터 손실, 보안 결함
- **major**: 기능 저하, UX 심각 훼손, AC 부분 미충족
- **minor**: 코드 품질/스타일

### C3. Active Evaluation 의무 (베이스라인)

- Sprint Contract Done Criteria의 모든 항목을 코드에서 직접 증명
- "구현됨" 진술만으로 PASS 금지 — logic tracing 필수
- Skepticism: "버그가 있다고 가정하고 찾는다"

### C4. Pressure 모드 처리 (베이스라인)

- 🟡 Caution: Minor 이슈는 보고하되 verdict에 영향 없음
- 🔴 Urgent: Critical 이슈만 검증

### C5. Cross-path Cleanup/Rollback 검증 (v2 신규)

> Source: free-tab-diversification / group-002 Major (BR-2 legacy rollback 누락)

**원칙**: 생성·쓰기 파이프라인에 복수 진입 경로(legacy vs new workflow, sync vs async 등)가 존재하면, 실패 처리(rollback/cleanup) 로직이 **모든 경로에서 동일하게 호출**되는지 검증한다.

**검증 포인트**:
- 신규 workflow/service에만 rollback이 추가되고 legacy 경로가 누락되었는가?
- 공통 helper(예: `rollbackXyzByContentId`)로 일원화되어 있는가, 또는 각 진입점에서 중복 구현되었는가?
- Pre-existing 에러 핸들러(`setContentError` 등)에 신규 cleanup 의무가 주입되어 있는가?

### C6. Pricing/Display Source 일관성 (v2 신규)

> Source: free-tab-diversification / group-002 Major (어제 폴백 pricing 과다 노출)

**원칙**: 서버 응답의 pricing 필드는 실제 생성 시 서버가 계산하는 eligibility와 일치해야 한다. 폴백/스냅샷 응답도 동일 규칙.

**검증 포인트**:
- UI 표시용 데이터와 pricing 계산용 데이터가 명시적으로 분리되어 있는가?
- 폴백 응답에서 `requiredCredit` 등 pricing 필드가 실제 eligibility와 일치하는가?
- Client가 표시하는 "무료/유료" 배지가 실제 생성 경로의 분기와 어긋날 가능성이 없는가?

### C7. Route Params 전달 완전성 & 타입 변경 시 Callsite (v2 신규 + v3 확장)

> Source: free-tab-diversification / group-003 Major (freeUsedToday passthrough 누락) + ugc-platform-001 / group-002 Major (Home 타입 변경 후 `tag: "all"` legacy callsite regression)

**원칙**: `route.types.ts`(또는 navigation 타입 정의)에 **신규 param 추가** 또는 **기존 param 타입 변경** 시, 모든 `navigation.navigate` / `navigation.dispatch` 호출부에서 호환 전달을 유지하고 `useCallback`/`useMemo` deps에도 포함해야 한다.

**검증 포인트**:
- grep으로 모든 `navigation.navigate("ScreenName"` 호출부 수집 → 신규 param 전달 여부 확인.
- `useCallback` deps 배열에 신규 param이 포함되어 있는가 (stale closure 회피).
- Deep link 경로 등 params가 없는 진입에서도 기본값/fallback이 동작하는가.
- **(v3 추가)** 기존 param 의 타입 변경 (e.g. `{ tag?: string }` → `NavigatorScreenParams<...>`) 시 legacy callsite 전수 점검. FE typecheck 을 `grep -v '@wrtn/'` 로 clean 측정해 pre-existing cascade 와 신규 regression 구분.

### C8. Deep Link 경로 Passthrough 부재 방어 (v2 신규)

> Source: free-tab-diversification / group-003~004 Minor 반복 패턴

**원칙**: 새 화면이 navigation params에 의존하는 데이터를 가질 때, deep link 직진입 경로에서도 최소 동작(기본값 렌더 또는 fetch fallback)을 보장해야 한다.

**검증 포인트**:
- Deep link handler에 필수 param 누락 대응이 있는가?
- Params 없을 때 fetch fallback이 존재하거나 UI가 "준비 중"/"로딩" 상태로 degrade 되는가?
- E2E에서 deep link 직진입 케이스가 커버되는가 (optional assertion 허용).

### C9. E2E 환경 의존성 명시 (v2 신규)

> Source: free-tab-diversification / group-003~004 E2E 환경 의존 fail

**원칙**: E2E flow는 seed fetch / token refresh / VPN 의존 실패 시 graceful degradation 가능해야 하며, 요구되는 환경 조건이 flow 상단 주석 또는 문서에 명시되어야 한다.

**검증 포인트**:
- seed-dependent assertVisible에 `optional: true` 적용 여부.
- 환경 prerequisites (VPN/토큰/시드)가 flow 상단 주석에 명시되어 있는가?
- CI에서 env 부재 시 skip/warn 처리가 가능한가?

### C10. Cursor 페이지네이션 $lte 규약 (v3 신규)

> Source: ugc-platform-001 / group-001 Major (be-003/004 `$lt` 사용으로 page 2 마지막 항목 누락)

**원칙**: CursorResponseDto 규약 — Service 가 `list[limit]` extra 항목 id 를 cursor 로 인코딩 — 에 정합하도록, Repository cursor 쿼리는 반드시 `_id: { $lte: cursorId }` 를 사용한다. `$lt` 는 page+1 에서 extra 항목이 조회되지 않아 마지막 아이템 누락.

**검증 포인트**:
- `rg '_id:\s*\{\s*\$lt\s*:' src/persistence/**/*.repository.ts` → 0 hit
- 동일 codebase 의 선례 (예: `filter.repository.ts`) 의 쿼리 연산자와 일치 확인
- E2E 에서 limit=2 + seed 3개 → page 2 list.length=1 assert

### C11. Nx e2e-spec testMatch / Harness 검증 (v3 신규)

> Source: ugc-platform-001 / group-001 Major (me-contents / me-profile / users-public e2e-spec 3개가 default testMatch 미매치로 0회 실행)

**원칙**: Nx monorepo 에서 신규 `.e2e-spec.ts` 추가 시, 단순 파일 존재 = PASS 가 아니다. `project.json` 의 `test-e2e` target + `jest-e2e.json` 의 `moduleNameMapper` 가 신규 파일을 discover 하고 실행하는지 증거로 확보해야 한다.

**검증 포인트**:
- `nx test {project} --listTests | grep e2e-spec` → 신규 파일 포함 여부
- project.json 에 `test-e2e` target 존재 + config 가 `jest-e2e.json` 가리킴
- jest-e2e.json 의 `moduleNameMapper` 가 path alias (`^src/(.*)$` 등) 포함
- Evaluator 는 파일 내용 read 가 아닌 **실행된 test 수** 를 증거로 요구

### C12. Discriminated Union / Parent-dependent Query Enabled Gate (v3 신규)

> Source: ugc-platform-001 / group-003 Major (SwipeFeed 양쪽 queryFn 무조건 fire) + Minor (OtherUserProfile contents query profile 미도착 시 fire)

**원칙**: Discriminated union route param 으로 variant 별 queryFn 을 선택하거나, parent query 결과에 의존하는 child query 를 구성할 때, 각 react-query hook 은 반드시 `enabled` option 으로 fire 제어해야 한다. Hook 은 hooks-rule 준수 위해 unconditional 호출하되 실행 여부는 `enabled` 로 결정.

**검증 포인트**:
- Discriminated union params 의 경우 `{ enabled: isVariantA }` 가드
- Parent-dependent query 는 `enabled: !!parentData` 가드
- Network tab / test mock 으로 "경로 분기 전" unintended fire 확인
- Contract §Done Criteria 에 variant 별 gate 명시 여부

### C13. Parent Context Prop Threading (v3 신규)

> Source: ugc-platform-001 / group-003 Major (ProfileContentItem 이 currentTab 을 모른 채 navigate payload 조립 요구)

**원칙**: 자식 컴포넌트의 navigate / action payload 가 parent context (현재 탭, 선택된 아이템, 현재 모드 등) 에 의존하는 경우, 해당 state 의 prop threading 경로를 Sprint Contract Done Criterion 에 명시한다. 자식이 parent state 를 추정 / 기본값 / hardcoding 으로 대체하는 것은 contract violation 으로 간주.

**검증 포인트**:
- Contract 의 navigate payload 가 parent state 를 참조하는지 먼저 식별
- 해당 state 가 callsite 컴포넌트의 props / context 에 존재하는지 역추적
- "자식이 parent state 를 추정/기본값으로 대체" 패턴 탐지
- Evaluator contract review (Round 1) 에서 Prop flow 수동 trace
